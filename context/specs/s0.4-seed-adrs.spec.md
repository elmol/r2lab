# Spec: S0.4 — Seed ADRs

## Goal

Capture the 6 architectural decisions already made during Phase 3 as ADRs, so future development sessions have context.

## What to Build

### 6 ADR files in `docs/adr/`

Each follows the template:

```markdown
# ADR-NNNN: [Title]

## Status
Accepted

## Context
[What problem or decision?]

## Decision
[What was decided and why?]

## Consequences
[Positive and negative consequences]

## Alternatives Considered
[What else was evaluated and why rejected?]
```

### ADRs to create

| File | Decision | Key rationale |
|------|----------|---------------|
| `0001-monorepo-flat-structure.md` | Flat monorepo (device/, attester/, common/, contracts/, webapp/) | Cohesion by responsibility, shared Cargo workspace |
| `0002-secp256k1-over-ed25519.md` | secp256k1/ECDSA for device identity | EVM-native, ecrecover in Solidity, wallet tooling compatible |
| `0003-alloy-for-evm-bindings.md` | Alloy over ethers-rs for Rust-EVM bindings | Active maintenance, `sol!` macro consumes Foundry ABI directly |
| `0004-single-contract-registry.md` | Single contract for registry + attestation | MVP simplicity, avoids cross-contract complexity |
| `0005-hybrid-storage.md` | Registration on-chain, data readings off-chain | Cost: high-frequency readings are prohibitive on-chain |
| `0006-emulation-mode.md` | Emulation mode for CI without physical RPi | Simulated serial + temperature, same signing logic |

### Source

Content is extracted from `context/architecture.md` (Sections 2, 4.5, 6.1, 7.4). No new decisions — just formalizing what's already decided.

## What NOT to Build

- ADR generation skill (nice-to-have, not blocking)
- Any new architectural decisions
