# S0.15: Versioning Workflow â€” cargo-release + Workspace Version

**Type:** Infrastructure â€” release best practices
**Depends on:** S0.12

---

## Problem

The version in `Cargo.toml` is hardcoded and never updated when a release tag is pushed. This means:
- `device --version` and `attester --version` always report `0.1.0` regardless of the actual release
- `Cargo.toml` and the git tag are permanently out of sync after the first version bump

---

## Solution

Two complementary changes:
1. **Workspace version** â€” one version declaration for all crates
2. **`cargo-release`** â€” one command to bump version, commit, tag, and push

Developer release workflow becomes:
```bash
cargo release minor   # 0.1.0 â†’ 0.2.0: bumps Cargo.toml, commits, tags v0.2.0, pushes
```
GitHub Actions triggers on the tag, compiles binaries with the correct version embedded.

---

## What to Build

### 1. Consolidate version in workspace `Cargo.toml`

Add `[workspace.package]` to the root `Cargo.toml`:

```toml
[workspace.package]
version = "0.1.0"
edition = "2021"
```

Read the existing root `Cargo.toml` first â€” only add what isn't already there.

### 2. Update each crate's `Cargo.toml` to inherit workspace version

In `device/Cargo.toml`, `attester/Cargo.toml`, and `protocol/Cargo.toml`, change:

```toml
# Before:
[package]
name = "device"
version = "0.1.0"
edition = "2021"

# After:
[package]
name = "device"
version.workspace = true
edition.workspace = true
```

### 3. Create `release.toml` at repo root

```toml
# cargo-release configuration for HardTrust
# Usage: cargo release minor (or patch, major, or specific version)

# Do not publish to crates.io
publish = false

# Tag format â€” must match release.yml trigger: "v*.*.*"
tag-name = "v{{version}}"
tag-message = "HardTrust {{version}}"

# Commit message for the version bump commit
pre-release-commit-message = "chore: release {{version}}"

# Run tests before releasing
pre-release-hook = ["cargo", "test", "--workspace"]

# Push commit + tag automatically
push = true
```

### 4. Add version guard to `.github/workflows/release.yml`

In **each** of the 3 build jobs (`build-device-armv7`, `build-attester-linux`, `build-attester-macos`), add this step immediately after `actions/checkout@v4`:

```yaml
- name: Verify Cargo version matches tag
  run: |
    CARGO_VERSION=$(cargo metadata --no-deps --format-version 1 \
      | python3 -c "import sys,json; pkgs=json.load(sys.stdin)['packages']; \
        print(next(p['version'] for p in pkgs if p['name']=='device'))")
    TAG_VERSION="${RELEASE_VERSION#v}"
    if [ "${CARGO_VERSION}" != "${TAG_VERSION}" ]; then
      echo "ERROR: Cargo.toml version (${CARGO_VERSION}) does not match tag (${TAG_VERSION})"
      echo "Use 'cargo release' to cut releases, not 'git tag' directly."
      exit 1
    fi
    echo "Version check OK: ${CARGO_VERSION}"
```

Note: `forge-build` is not needed for this step â€” `cargo metadata` works without building contracts.

## Developer Release Workflow (post-implementation)

```bash
# Dry run first â€” see what will happen
cargo release minor --dry-run

# Cut the release
cargo release minor    # 0.1.0 â†’ 0.2.0
# or
cargo release patch    # 0.1.0 â†’ 0.1.1
# or
cargo release 0.3.0    # specific version
```

cargo-release will:
1. Run `cargo test --workspace`
2. Bump `version` in workspace `Cargo.toml` (one place)
3. Commit: `chore: release 0.2.0`
4. Tag: `v0.2.0`
5. Push commit + tag â†’ GitHub Actions triggers â†’ binaries compiled with `0.2.0`

## What NOT to Build

- Do not install `cargo-release` in CI (it runs on developer machines only)
- Do not migrate to `cargo-dist`
- Do not change `build-device.sh` or `build-attester.sh`

## Validation Criteria

- [ ] `cargo metadata --no-deps --format-version 1` shows all 3 crates with same version
- [ ] `cargo release patch --dry-run` runs without error
- [ ] After cutting a release with `cargo release`, `device --version` reports the new version
- [ ] Pushing a tag manually without bumping Cargo.toml fails the CI version guard
- [ ] `just ci` passes
