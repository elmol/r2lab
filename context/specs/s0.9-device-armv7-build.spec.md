# S0.9: Device Binary — ARMv7 Cross-Compilation for Raspberry Pi

**Type:** Infrastructure / Build
**Story Reference:** S1a.1, S1b.1, S1c.1 (device binary must run on RPi)
**Depends on:** All S1 specs Implemented

---

## Target Platform

| Property | Value |
|----------|-------|
| OS | Raspbian GNU/Linux 11 (Bullseye) |
| Architecture | ARMv7 32-bit (armv7l) |
| Rust target triple | `armv7-unknown-linux-musleabihf` |
| Binary type | Fully static (musl) — no glibc dependency |

**Why musl?** Bullseye ships glibc 2.31. A glibc-linked binary built on a newer host (glibc 2.35+) will fail with `GLIBC_2.33 not found`. Musl produces a fully static binary that runs on any ARMv7 Linux regardless of glibc version.

---

## What to Build

### 1. Add Zig + cargo-zigbuild to the build environment

`cargo-zigbuild` uses the Zig compiler as a cross-linker. No Docker needed. `device` has zero C dependencies (pure Rust: clap, k256, serde_json, chrono) so this is the simplest possible cross-compilation case.

Install (one-time, developer machine or CI):
```bash
# Install Zig (via snap, apt, or direct download)
snap install zig --classic --beta
# OR: download from ziglang.org/download and add to PATH

# Install cargo-zigbuild
cargo install cargo-zigbuild

# Add the Rust target
rustup target add armv7-unknown-linux-musleabihf
```

### 2. Build the device binary

From the HardTrust workspace root:

```bash
cargo zigbuild \
  --target armv7-unknown-linux-musleabihf \
  --release \
  -p device
```

Output: `target/armv7-unknown-linux-musleabihf/release/device`

Expected binary size: ~3-8 MB stripped, ~15-25 MB unstripped.

To strip debug symbols (reduces size):
```bash
# Using the ARM strip bundled with cargo-zigbuild / zig
llvm-strip target/armv7-unknown-linux-musleabihf/release/device
# OR: just use the rust strip profile (add to Cargo.toml):
```

Add to workspace `Cargo.toml` under `[profile.release]`:
```toml
[profile.release]
strip = true
opt-level = 3
lto = true
codegen-units = 1
```

### 3. Add a justfile recipe

Add to `justfile`:

```justfile
# Build device binary for Raspberry Pi (ARMv7 musl — static binary, runs on Raspbian Bullseye)
build-rpi:
    cargo zigbuild --target armv7-unknown-linux-musleabihf --release -p device
    @echo "Binary: target/armv7-unknown-linux-musleabihf/release/device"
    @ls -lh target/armv7-unknown-linux-musleabihf/release/device
```

### 4. Copy to RPi and verify

After building, copy to the RPi:

```bash
# Replace PI_HOST with your RPi IP or hostname
scp target/armv7-unknown-linux-musleabihf/release/device pi@PI_HOST:~/device

# On the RPi:
chmod +x ~/device
./device --help
./device init
./device emit
```

Expected output of `./device --help`:
```
HardTrust device CLI — generate key and emit readings

Usage: device <COMMAND>

Commands:
  init  Initialize device: generate a secp256k1 key pair
  emit  Emit a signed temperature reading to stdout
```

## What NOT to Build

- Do not add GitHub Actions release workflow (that's S0.10)
- Do not change any Rust source code
- Do not cross-compile `attester` (not needed for RPi)
- Do not add Docker

## Validation Criteria

- [ ] `just build-rpi` succeeds on the developer machine
- [ ] `file target/armv7-unknown-linux-musleabihf/release/device` reports `ELF 32-bit LSB executable, ARM`
- [ ] Binary copied to RPi and `./device --help` prints usage
- [ ] `./device init` creates `~/.hardtrust/device.key` on the RPi
- [ ] `./device emit` outputs valid JSON with temperature reading
- [ ] `./device emit` shows `[EMULATED]` (no real hardware sensor on this target in test setup)
