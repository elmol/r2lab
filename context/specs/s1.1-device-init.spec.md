# Spec: S1.1 — Device Init (Generate Device Identity)

## Story Reference

[S1.1 — Generate Device Identity](../stories/slice-1/s1.1-generate-device-identity.md)

## Related ADRs

- ADR-0002: secp256k1 over ed25519 — why we use secp256k1 for device identity
- ADR-0001: Monorepo flat structure — why device/ and common/ are workspace members
- ADR-0006: Emulation mode — why and how `--emulate` works

## Goal

Implement `device init` — the first command that creates a secp256k1 keypair tied to the device's hardware serial.

## What to Build

### device/ crate (new binary)

Add `device` as a workspace member. CLI with one subcommand: `init`.

**`device init [--emulate]`:**

1. Read hardware serial:
   - Real: `/sys/firmware/devicetree/base/serial-number`
   - `--emulate`: hardcoded `"EMULATED-0001"`
2. Check if `~/.hardtrust/device.key` exists → if yes, print warning with existing address and exit
3. Generate secp256k1 keypair (random, OS entropy)
4. Derive Ethereum address from public key
5. Save private key to `~/.hardtrust/device.key` (dir 0o700, file 0o600)
6. Print:
   ```
   Device identity generated.
   Serial:  <serial>
   Address: 0x<address>
   ```

### common/ crate (new library)

Add `common` as a workspace member. Only the crypto functions needed for this story:

- Generate secp256k1 keypair
- Derive Ethereum address from public key
- Serialize/deserialize private key to/from bytes

Pure library — no I/O, no CLI.

### Error cases

- Serial not readable → suggest `--emulate`
- Can't write key file → clear error + exit 1
- Private key never printed to console

## What NOT to Build

- No `attester/` crate
- No contract interaction
- No signing/verification (that's S1.5/S1.6)
- No JSON output format
- No key rotation or force overwrite

## Validation

- `device init --emulate` prints serial + valid Ethereum address
- Second run detects existing key and aborts
- `just ci` passes (tests + lint)
