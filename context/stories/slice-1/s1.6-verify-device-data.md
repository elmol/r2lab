# S1.6 — Verify Device Data via CLI

## User Story
As anyone, I run the verify command with a signed JSON file so that I can check whether the data was emitted by a registered device or not.

## Acceptance Criteria

**AC-1: Verify data from a registered device**
- Given a signed JSON reading emitted by a device that IS registered on-chain
- When the user runs the verify command pointing to the JSON file
- Then the CLI recovers the signer address from the signature
- And queries the contract for registration status
- And prints: **VERIFIED** — device is registered, signature is valid

**AC-2: Verify data from an unregistered device**
- Given a signed JSON reading emitted by a device that is NOT registered on-chain
- When the user runs the verify command
- Then the CLI recovers the signer address
- And queries the contract
- And prints: **UNVERIFIED** — device is not registered

**AC-3: Tampered data detected**
- Given a JSON file where the payload has been modified after signing
- When the user runs the verify command
- Then the signature verification fails
- And prints: **INVALID** — signature does not match the data

## Edge Cases

| Scenario | Expected Behavior |
|----------|-------------------|
| JSON file not found or malformed | Clear error indicating the file could not be read |
| RPC endpoint not reachable | Clear error indicating connection failure |
| JSON contains multiple readings | Verify each reading individually, report result per reading |

## Definition of Done
- `attester verify --file <path>` returns VERIFIED, UNVERIFIED, or INVALID correctly
- All three cases are covered by automated tests against Anvil
- Tests pass in CI using emulation mode
- Linting and formatting pass
- **The full Slice 1 demo is runnable:** init → register → emit → verify (VERIFIED) + unregistered device emit → verify (UNVERIFIED)
