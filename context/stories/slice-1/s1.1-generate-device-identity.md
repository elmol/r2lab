# S1.1 — Generate Device Identity

## User Story
As a Device Owner, I run the init command on my RPi so that the device generates a unique identity (keypair) tied to its hardware serial.

## Acceptance Criteria

**AC-1: First-time identity generation**
- Given a device with no existing identity
- When the Device Owner runs the init command
- Then the system reads the hardware serial number, generates a secp256k1 keypair, persists the private key to filesystem, and prints the serial and derived Ethereum address to console in a clearly labeled format

**AC-2: Existing keys detected**
- Given a device that already has a persisted keypair
- When the Device Owner runs the init command
- Then the system warns that keys already exist and does NOT overwrite them

**AC-3: Emulation mode**
- Given a CI or development environment without a physical RPi
- When the init command is run with emulation mode activated
- Then the system uses a hardcoded simulated serial number and behaves identically to the real-hardware path

## Edge Cases

| Scenario | Expected Behavior |
|----------|-------------------|
| Serial file does not exist or is unreadable | Clear error, suggest emulation mode |
| Filesystem is read-only or key can't be persisted | Clear error, exit |

## Dependencies
- **Upstream:** None (first story)
- **Downstream:** S1.3 (register on-chain), S1.5 (emit data)
- **Shared:** `common/` crate (first consumer of secp256k1 key generation)

## Out of Scope
- Key overwrite/force flag
- JSON output format
- Configurable data directory
- Key rotation, encrypted storage
- QR code output
- Any on-chain interaction

## Definition of Done
- `device init` runs in emulation mode and produces the expected console output
- Existing keys are detected and not overwritten
- Tests pass in CI (emulation mode)
- Linting and formatting pass
- No secrets leaked to console output
