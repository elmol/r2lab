# S1.3 — Register Device via CLI

## User Story
As an Attester, I register a device on-chain using the CLI so that the device is recognized as attested in the network.

_Note: The Attester receives the device's serial and Ethereum address from the Device Owner by any means (chat, email, in person) before running this command._

## Acceptance Criteria

**AC-1: Successful registration**
- Given a deployed contract with the caller as an authorized attester
- When the Attester runs the register command with a device serial and Ethereum address
- Then a registration transaction is submitted on-chain
- And the transaction hash and confirmation status are printed to console
- And the device is queryable as registered in the contract

**AC-2: Duplicate registration rejected**
- Given a device serial that is already registered
- When the Attester tries to register the same serial again
- Then the transaction reverts
- And a clear error message is shown

**AC-3: Unauthorized caller rejected**
- Given a wallet that is NOT an authorized attester
- When it tries to register a device
- Then the transaction reverts
- And a clear error message is shown

## Edge Cases

| Scenario | Expected Behavior |
|----------|-------------------|
| Invalid serial format | Clear error before submitting tx |
| Invalid Ethereum address format | Clear error before submitting tx |
| RPC endpoint not reachable | Clear error indicating connection failure |

## Dependencies
- **Upstream:** S1.2 (contract deployed + attester seeded)
- **Downstream:** S1.4 (query status), S1.5 (emit), S1.6 (verify)
- **Shared:** `common/` (ABI bindings, types), `attester/` crate

## Out of Scope
- Batch registration (R2.1)
- Revocation (R2.3)
- Web portal registration (S2.2)
- Physical verification workflow (that's a human step, not system behavior)

## Definition of Done
- `attester register --serial <serial> --address <address>` succeeds against Anvil
- Duplicate and unauthorized cases are tested
- Tests pass in CI
- Linting and formatting pass
