# S1.5 — Emit Single Signed Reading to JSON

## User Story
As a Device Owner, I run the emit command on my RPi so that it reads the CPU temperature, signs the reading with the device's private key, and saves it to a local JSON file.

## Acceptance Criteria

**AC-1: Emit a signed reading**
- Given a device with an existing identity (keypair from S1.1)
- When the Device Owner runs the emit command
- Then the system reads the CPU temperature and current timestamp
- And signs the payload with the device's private key
- And writes the signed reading to a local JSON file
- And prints confirmation to console

**AC-2: Emulation mode**
- Given a CI/development environment without physical RPi
- When the emit command runs in emulation mode
- Then a simulated temperature value is used
- And the rest of the flow (sign, save) behaves identically

**AC-3: No identity found**
- Given a device that has NOT run init (no keypair exists)
- When the emit command is run
- Then a clear error indicates that the device must be initialized first

## Edge Cases

| Scenario | Expected Behavior |
|----------|-------------------|
| JSON file already exists with previous readings | Append the new reading, don't overwrite |
| Filesystem is read-only | Clear error indicating the reading could not be saved |

## Dependencies
- **Upstream:** S1.1 (device identity must exist for signing)
- **Downstream:** S1.6 (verify reads the JSON file)
- **Shared:** `common/` (signing logic, payload types), `device/` crate

## Out of Scope
- Continuous/periodic emission (S3.4)
- Sending data to an API (Slice 2)
- IPFS storage (future)
- Configurable emission interval

## Definition of Done
- `device emit` produces a signed JSON file in emulation mode
- Missing identity case is handled with clear error
- JSON structure contains: serial, temperature, timestamp, signature
- Tests pass in CI
- Linting and formatting pass
