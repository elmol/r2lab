# S1.2 — Deploy Contract + Seed Attester

## User Story
As a developer/deployer, I deploy the HardTrust registry contract to a local chain (Anvil) with at least one pre-registered attester so that the system is ready to register devices.

## Acceptance Criteria

**AC-1: Contract deploys successfully**
- Given Anvil is running locally
- When the deploy script is executed
- Then the registry contract is deployed and the contract address is printed to console

**AC-2: Attester is pre-seeded**
- Given a successful deployment
- When the deploy script completes
- Then at least one Ethereum address is registered as an authorized attester in the contract

**AC-3: Contract is functional**
- Given a deployed contract with a seeded attester
- When the attester address is queried
- Then the contract confirms it is an authorized attester

## Edge Cases

| Scenario | Expected Behavior |
|----------|-------------------|
| Anvil is not running | Clear error indicating no RPC connection |
| Deploy script run twice | Deploys a new instance (no conflict on local chain) |

## Dependencies
- **Upstream:** None (can be done in parallel with S1.1)
- **Downstream:** S1.3 (register device), S1.4 (query), S1.5 (emit), S1.6 (verify)
- **Shared:** `contracts/` (Foundry project), deploy script in `contracts/script/`

## Out of Scope
- Multiple attesters
- Contract upgradability
- Deploy to Sepolia (that's release/infra concern)
- Owner transfer

## Definition of Done
- `forge script` deploys the contract to Anvil
- At least one attester address is seeded
- Foundry tests pass for the contract (add/remove attester, basic state)
- CI can run deploy + tests against Anvil
